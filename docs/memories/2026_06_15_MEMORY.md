# Session Memory — 2026-06-15

## Summary
Built an MCP server so external AI models can read/modify Flowmancer boards and
tickets. Lives in a self-contained Node package at `mcp-server/` (separate from
the Vue app). Chosen design: MCP + shared core, API-key -> service-account auth.

## Why a separate package
The web app is frontend-only (Firebase client SDK, static SPA on Vercel, no
backend). External AI can't safely use the public client config (gated by
per-user Auth), so the MCP server uses the **Firebase Admin SDK** with a
service account to act on Firestore directly.

## Structure (`mcp-server/`)
- `src/firebase.js` — Admin SDK init from `FIREBASE_SERVICE_ACCOUNT` (JSON
  string) or `GOOGLE_APPLICATION_CREDENTIALS` (file path) + `FIREBASE_PROJECT_ID`.
- `src/board-service.js` — Core ops, mirrors `src/stores/board.js` exactly
  (same `generateId`, same nested columns/cards array model, card timestamps as
  `Date.now()` numbers, board `updatedAt` as serverTimestamp). Exposes:
  `resolveApiKey`, `listBoards`, `getBoard`, `createTicket`, `moveTicket`,
  `updateTicket`, `assignTicket`, `deleteTicket`. Enforces the same permission
  model: member must be `owner`/`editor` to mutate; viewers read-only.
- `src/server.js` — MCP server (stdio) using `@modelcontextprotocol/sdk` v1.29
  `registerTool` + zod input schemas. Tools: `whoami`, `list_boards`,
  `get_board`, `create_ticket`, `move_ticket`, `update_ticket`,
  `assign_ticket`, `delete_ticket`. API key read from env `FLOWMANCER_API_KEY`.
- `src/errors.js` — `ApiError { message, code }` for clean tool error output.
- `scripts/create-api-key.js` / `revoke-api-key.js` — manage keys. Keys are
  `fmk_live_<hex>`, only the SHA-256 **hash** is stored in a new `apiKeys`
  Firestore collection; plaintext printed once. Map by `--uid` or `--email`.

## Auth model
API key -> `apiKeys` doc { hash, uid, name, scopes:[read,write], revoked } ->
caller acts as that Firebase uid. Columns/cards resolvable by id OR name
(case-insensitive) so the AI can say "move to In Progress".

## Firestore rules — SECURITY FIX (deployed 2026-06-15)
- Discovered the LIVE rules had a Firebase test-mode catch-all
  `match /{document=**} { allow read, write: if true }` making the ENTIRE
  database public (read+write, even unauthenticated). The specific boards/users
  rules below it were dead code (rules are OR-additive).
- Fetched live rules via the Firebase Rules REST API using the service-account
  access token (admin SDK `credential.getAccessToken()`), backed them up to
  `firestore.rules.live-backup` (repo root).
- Wrote corrected `firestore.rules` + `firebase.json` + `.firebaserc` in repo
  root. Removed the catch-all; kept boards/users as designed; ADDED a `projects`
  rule (per-user, `userId == request.auth.uid` — it previously only worked via
  the catch-all, NO specific rule existed); added `apiKeys` deny-all.
- Boards read rule got a `userId` fallback (owner can always read, helps legacy
  backfill) before the `members` check (short-circuits to avoid errors when a
  legacy doc lacks `members`).
- DEPLOYED via Rules API: POST ruleset, then PATCH release
  `projects/<p>/releases/cloud.firestore` with body
  `{ release: { name, rulesetName } }` (note the `release` wrapper — bare
  `{rulesetName}` returns 400). Service account HAS publish permission.
- Verified live: no `if true`, no wildcard, apiKeys/projects present. MCP server
  (admin SDK, bypasses rules) confirmed still working after deploy.

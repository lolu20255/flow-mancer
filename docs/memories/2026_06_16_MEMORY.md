# Session Memory: 2026-06-16

## Focus
Built the **Agents Monitor**: a live Dashboard view showing which repos currently
have an AI agent (Claude Code, Codex) working, so the user (who runs 10-15 apps
with agents) can see at a glance which are still running.

## What was built

### MCP server (`mcp-server/`)
- **`src/agent-service.js`** (new): per-session presence in Firestore
  `agentSessions` collection. `startAgentSession` / `beatAgentSession` /
  `stopAgentSession` / `listAgentSessions` + `resolveSessionProject`. Doc id is
  `${agent}__${sessionId}`. Start is idempotent (keeps `startedAt`); beat uses
  `update` so it never resurrects a stopped session; stop deletes the doc.
- **`scripts/agent-hook.js`** (new): the CLI the hooks call (`start|beat|stop`).
  Reads hook JSON from stdin, finds the repo's `.flowmancer` (walks upward),
  self-loads `mcp-server/.env` and absolutizes the credentials path, caches the
  resolved uid in tmp, throttles heartbeats ~45s. Always exits 0 (fail-safe).
  **Opt-in guard**: if no `.flowmancer` (and no project flag), it exits early and
  writes nothing, so the global hooks stay silent in untracked repos.
- **`src/server.js`**: added `list_projects` and `list_agent_sessions` MCP tools.
- **`package.json`**: added `flowmancer-agent` bin + `agent` script.
- **`scripts/deploy-rules.js`** (new): deploys `firestore.rules` via the Firebase
  Rules REST API using the service-account token (POST ruleset, PATCH release).
- **`.flowmancer.example`** + README "Agents monitor" section.

### Web app (`src/`)
- **`stores/agents.js`** (new): `onSnapshot` on `agentSessions where userId==uid`,
  groups sessions per project, `STALE_MS = 20min` stale threshold.
- **`components/AgentMonitor.vue`** (new): per-project cards with a pulsing green
  "working" ping, task summary, elapsed time, agent badge, "stale" badge.
- **`views/Dashboard.vue`**: renders `<AgentMonitor />` above Projects (only shows
  when sessions exist).
- **`App.vue`**: inits/cleans up the agents store alongside the others.

### Firestore rules
- Added `agentSessions` rule: owner-read, write denied (Admin-SDK only).
- **Deployed live** to `vibe-board-3b2cf` via `deploy-rules.js`.

### Hooks (user global `~/.claude/settings.json`)
- Wired `UserPromptSubmit` -> `start` and `Stop` -> `stop`.
- Initially also added `PreToolUse` -> `beat` but **removed it**: it spawned a
  node process on every tool call and felt like it was blocking long turns.
  Start/stop-only is the trade-off behind the 20-min stale window.
- `.flowmancer` files created: `~/Development/Apps/sourdough/sourdough-app`
  (SourDoughApp) and `~/Development/Apps/s3-sync` (SnapVault).

## Key facts / decisions
- App uses **Firestore** (not Realtime Database). Live updates via `onSnapshot`;
  RTDB's onDisconnect doesn't help because hooks are one-shot processes.
- Project name -> id map (from `projects` collection): ReadLens
  `w7Xk3x4nALxfKI2aPYny`, SourDoughApp `Dcv6qnNG7r8YtbkWOOK1`, SnapVault
  `AFQD4wmGQcKNIQ110HOv`, etc.
- A 529 API error during testing was Anthropic overload, NOT the hooks.

## Open follow-ups
- Hooks load at session start; user must restart Claude Code (and approve the
  changed hooks) for the slimmer config to apply.
- `.flowmancer` is searched UPWARD only: running Claude from a parent dir above
  it (e.g. `~/Development/Apps/sourdough`) won't trigger tracking. User was asked
  whether to add a `.flowmancer` at the parent level (unanswered).
- New MCP tools (`list_projects`, `list_agent_sessions`) require restarting the
  MCP client to appear.
- If a turn dies (e.g. 529) in a tracked repo, `Stop` may not fire and the card
  lingers until the 20-min stale timeout or the next turn clears it.
- Optional: move hooks from global to per-repo `.claude/settings.json` if global
  overhead is ever a concern.
